1. How the Build Chain Works

The original project uses a two-stage Docker build:

Stage 1: BASE IMAGE (Ubuntu + system packages) ┌───────────────────────────────────────────────────────────────┐ │ docker/dockerfile-noble.base │ │ Ubuntu 24.04 + python3 + git + cmake + sqlite3 + smarttools │ │ ───────────────────────────────────────────────────────────── │ │ Pushed to: ghcr.io/guydavis/machinaris-base-noble:latest │ └──────────────────────────────┬────────────────────────────────┘ │ FROM (pulls) │ Stage 2: MAIN IMAGE (all blockchains + Machinaris app) ┌──────────────────────────────┴────────────────────────────────┐ │ docker/dockerfile │ │ Installs 34 blockchain forks + Machinaris WebUI/API + tools │ │ ───────────────────────────────────────────────────────────── │ │ Pushed to: ghcr.io/guydavis/machinaris:latest │ └───────────────────────────────────────────────────────────────┘
Key insight The base image is just Ubuntu + system packages — nothing blockchain-specific. You can keep using Guy Davis's base image and only build + push the main image under your own registry. This saves significant build time and is the recommended starting approach.

The main Dockerfile line 4 contains:

FROM ghcr.io/guydavis/machinaris-base-${UBUNTU_VER}:${MACHINARIS_STREAM}

With defaults UBUNTU_VER=noble and MACHINARIS_STREAM=main, this pulls ghcr.io/guydavis/machinaris-base-noble:main.

2. Choose Your Approach

A. GitHub Actions (Recommended)

  • Builds on GitHub's Linux runners
  • Supports multi-arch (amd64 + arm64)
  • Pushes directly to GHCR
  • No local Docker needed
  • Free for public repos
  • Automated on push/dispatch

B. Local Windows Build

  • Docker Desktop required
  • Linux containers mode
  • Builds amd64 natively
  • Push to any registry
  • Full control over build
  • Slow (installs 34 forks)

C. Build on Unraid

  • No registry needed
  • Image stays local
  • Native Linux build
  • Simple workflow

C. Drawbacks

  • Ties up Unraid CPU for hours
  • Needs git + build tools
  • Image not portable
  • Must rebuild on-box each time

3. Approach A — GitHub Actions CI/CD (Recommended)

This is how the original project built and published its images. You fork the repo, update the workflow files to use your registry, and GitHub builds and pushes the image for you.

1

Fork the repo to your GitHub account

If you haven't already, fork SquareMesh/machinaris to your own GitHub account (e.g., yourusername/machinaris). This gives you GitHub Actions runners and a GHCR namespace automatically.

2

Configure repository secrets

Go to your forked repo → Settings → Secrets and variables → Actions → New repository secret:

Secret NameValuePurpose
DOCKERHUB_USERNAMEYour DockerHub usernameUsed in image tags (if pushing to DockerHub too)
DOCKERHUB_TOKENDockerHub access tokenAuthentication for DockerHub push
GHCR is free GITHUB_TOKEN is provided automatically — no secret needed for GHCR. If you only want to use GHCR (not DockerHub), you can simplify the workflows to remove the DockerHub steps entirely.
3

Update the workflow files

Edit .github/workflows/main-blockchains.yaml to use your image names.

The key change is the tags: section. Replace guydavis with your username:

# .github/workflows/main-blockchains.yaml
# In the "Build and push" step, change tags to:
tags: |
  ghcr.io/yourusername/machinaris:latest
  ghcr.io/yourusername/machinaris:v${{ github.event.inputs.version }}

Also update the build-args to set the Chia version you want:

build-args: |
  "UBUNTU_VER=noble"
  "MACHINARIS_STREAM=latest"
  "CHIA_BRANCH=2.6.0"
  "BLADEBIT_BRANCH=master"
  "MADMAX_BRANCH=v2.5.5.giga36"
4

Ensure GHCR package visibility

By default, GHCR packages from public repos are public. If your repo is private, you'll need to make the package public or configure Unraid to authenticate with GHCR.

Go to your GitHub profile → Packages → find machinaris → Package settings → set visibility to Public.

5

Trigger the build

The release workflow uses workflow_dispatch (manual trigger):

  1. Go to your repo → Actions tab
  2. Select release-blockchains workflow
  3. Click Run workflow
  4. Enter the version number (e.g., 2.6.1)
  5. Click Run workflow

GitHub will build the image on their Linux runner and push it to GHCR. This takes 30–60+ minutes due to installing all blockchain forks.

6

Verify the image is published

# From any machine with Docker:
docker manifest inspect ghcr.io/yourusername/machinaris:latest

# Or check on GitHub: Packages tab on your profile
Result Your image is now at ghcr.io/yourusername/machinaris:latest and can be pulled by your Unraid server.

4. Approach B — Build Locally on Windows

Prerequisites
  • Docker Desktop for Windows installed and running in Linux containers mode (not Windows containers)
  • WSL 2 backend enabled (Docker Desktop settings)
  • At least 30 GB free disk space for the image build
  • Expect the build to take 1–3 hours depending on your machine
1

Verify Docker is in Linux mode

# Check Docker is running with Linux containers
docker info | grep "OSType"
# Should output: OSType: linux

# If it says "windows", right-click Docker Desktop tray icon
# and select "Switch to Linux containers"
2

Log in to your container registry

# For GitHub Container Registry (GHCR):
# First, create a Personal Access Token at:
# GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic)
# Scopes needed: write:packages, read:packages, delete:packages

echo YOUR_GITHUB_PAT | docker login ghcr.io -u yourusername --password-stdin

# For DockerHub:
docker login -u yourusername
3

Build the image

From your project directory (C:\0dev\ClaudeCodeProjects\machinaris-1):

# Option 1: Use the existing base image (recommended, much faster)
docker build ^
  -f docker/dockerfile ^
  -t ghcr.io/yourusername/machinaris:latest ^
  --build-arg UBUNTU_VER=noble ^
  --build-arg MACHINARIS_STREAM=latest ^
  --build-arg CHIA_BRANCH=2.6.0 ^
  .

# Option 2: Build your own base image first (full independence)
docker build ^
  -f docker/dockerfile-noble.base ^
  -t ghcr.io/yourusername/machinaris-base-noble:latest ^
  .

# Then build the main image FROM your base:
# (edit docker/dockerfile line 4 to point to your base image first)
docker build ^
  -f docker/dockerfile ^
  -t ghcr.io/yourusername/machinaris:latest ^
  --build-arg UBUNTU_VER=noble ^
  --build-arg MACHINARIS_STREAM=latest ^
  --build-arg CHIA_BRANCH=2.6.0 ^
  .
Build Args Only blockchain forks with a _BRANCH value will be installed. If you only care about Chia, you can set only CHIA_BRANCH=2.6.0 and leave the rest empty — the install scripts skip forks with no branch specified, making the build much faster.
4

Push to the registry

# Push to GHCR
docker push ghcr.io/yourusername/machinaris:latest

# Optionally tag with a version too
docker tag ghcr.io/yourusername/machinaris:latest ^
           ghcr.io/yourusername/machinaris:v2.6.1
docker push ghcr.io/yourusername/machinaris:v2.6.1
5

Verify the push

# Check it exists on the registry
docker manifest inspect ghcr.io/yourusername/machinaris:latest
Architecture note Building on Windows (Docker Desktop) produces a linux/amd64 image only. This works fine for Unraid (which runs x86_64). If you also need linux/arm64, use GitHub Actions (Approach A) which uses QEMU for cross-architecture builds.

5. Approach C — Build on Unraid Directly

Simplest workflow, but ties up Unraid resources. No registry needed — the image stays local.

1

SSH into Unraid and clone

ssh root@your-unraid-ip

# Clone to a persistent location
cd /mnt/user/appdata
git clone https://github.com/yourusername/machinaris.git machinaris-src
cd machinaris-src
2

Build the image locally

docker build \
  -f docker/dockerfile \
  -t machinaris-custom:latest \
  --build-arg UBUNTU_VER=noble \
  --build-arg MACHINARIS_STREAM=latest \
  --build-arg CHIA_BRANCH=2.6.0 \
  .

The image machinaris-custom:latest is now available locally on Unraid.

3

Use the local image name when creating the container

In the Unraid Docker UI, set Repository to machinaris-custom:latest (no registry prefix means local).

6. Deploy to Unraid

Once your image is available (either on a registry or built locally on Unraid), follow the steps in the Unraid Migration Guide to swap the container. The key step is simply changing the Repository field:

FieldOriginal ValueNew Value
Repositoryghcr.io/guydavis/machinaris:latestghcr.io/yourusername/machinaris:latest

Everything else (volumes, ports, environment variables) stays the same.

1

If using GHCR with a private repo

Unraid needs to authenticate to pull from private GHCR packages. SSH into Unraid and log in:

# On Unraid terminal:
docker login ghcr.io -u yourusername -p YOUR_GITHUB_PAT

# This saves credentials to /root/.docker/config.json
# Persists across reboots if on the flash drive
2

Pull and verify

# Pre-pull the image to verify it works before swapping
docker pull ghcr.io/yourusername/machinaris:latest

# Check it's there
docker images | grep machinaris
3

Swap the container

Follow the Unraid Migration Guide sections 3–5: stop old container → remove → create new with your image → verify.

7. Troubleshooting

ProblemCauseFix
Build fails: FROM ghcr.io/guydavis/machinaris-base-noble:main not found Base image tag mismatch Use --build-arg MACHINARIS_STREAM=latest (the tag that exists on GHCR), or build your own base image first.
docker build fails on Windows with permission errors Docker Desktop not in Linux containers mode Right-click Docker Desktop tray → "Switch to Linux containers". Ensure WSL 2 backend is enabled.
Build takes hours 34 blockchain forks are being installed Only set the _BRANCH args for forks you actually need. Unset branches are skipped. E.g., only CHIA_BRANCH=2.6.0 for Chia-only.
denied: permission_denied when pushing to GHCR PAT missing write:packages scope Create a new PAT with scopes: write:packages, read:packages. Re-login with docker login ghcr.io.
Unraid can't pull from GHCR Package is private Make the GHCR package public, or run docker login ghcr.io on Unraid first.
no matching manifest for linux/amd64 Image was built for wrong architecture Ensure you're building for linux/amd64. On Windows Docker Desktop this is the default.
GitHub Actions workflow not visible Workflows not on default branch Push workflow files to main branch. GitHub only shows workflows from the default branch in the Actions tab.
Summary Fastest path: Use GitHub Actions — push your code, trigger the workflow, it builds a linux/amd64 image and pushes to GHCR. Then on Unraid, just change the Repository field to ghcr.io/yourusername/machinaris:latest.

Local build path: Docker Desktop (Linux mode) on Windows can build it too. Build, push to GHCR, pull on Unraid. Takes longer but gives you full control.
