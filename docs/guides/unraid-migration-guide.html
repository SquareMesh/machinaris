Machinaris — Unraid Docker Migration Guide

Replace the original guydavis/machinaris container with your own forked image while preserving all farm data, plots, and configuration.

What this guide covers

  • Back up your current Machinaris configuration and database
  • Document your existing container settings (volumes, ports, env vars)
  • Stop and remove the original container
  • Build or pull your new forked image
  • Create a new container pointing to the same persistent data
  • Verify the migration was successful
Critical Your plots, wallet keys, and blockchain database live on persistent volumes, not inside the container. As long as you map the same volume paths, no data is lost when you swap containers. However, always back up your mnemonic key before making changes.

0. Prerequisites

Before you start

1. Document Your Current Setup

Before changing anything, record every setting from your running container. You will need these exact values when creating the replacement.

1

Open the Unraid Docker page

Navigate to the Docker tab in your Unraid WebUI. Find your Machinaris container in the list.

2

Export the container template (Unraid UI method)

Click on the Machinaris container icon, then select Edit. This shows all configured settings. Take a screenshot or copy down every field. Alternatively, from the Unraid terminal:

# Inspect the running container to capture all settings
docker inspect machinaris > /boot/config/machinaris-backup.json

# This saves: image name, volumes, ports, env vars, restart policy
3

Record critical settings

At minimum, capture these values:

SettingWhere to find itExample
Container nameDocker tabmachinaris
ImageEdit → Repositoryghcr.io/guydavis/machinaris:latest
Network typeEdit → Networkbridge or br0
Port mappingsEdit → Ports8926:8926, 8927:8927, 8444:8444
Volume mappingsEdit → PathsSee table below
Environment varsEdit → Variablesblockchains=chia, mode=fullnode, etc.
Extra paramsEdit → Extra ParametersGPU passthrough flags, etc.
4

Record all volume mappings

These are the most important settings — they point to your persistent data:

Container pathTypical Unraid host pathPurpose
/root/.chia/mnt/user/appdata/machinaris/chiaConfig, wallet keys, databases
/plots/mnt/user/plots (or multiple)Plot files
/plotting/mnt/cache/plotting (SSD/NVMe)Temp plotting space
Tip You may have multiple /plots and /plotting paths (e.g., /plots2, /plots3). Record every one. Run docker inspect machinaris | grep -A2 "Mounts" to see all mounts.
5

Back up your appdata

# Back up the entire Machinaris appdata directory
cp -r /mnt/user/appdata/machinaris /mnt/user/appdata/machinaris-backup-$(date +%Y%m%d)

# Most critically, back up your mnemonic key
cp /mnt/user/appdata/machinaris/chia/mnemonic.txt /boot/config/mnemonic-backup.txt
Critical Your mnemonic.txt file contains your wallet recovery phrase. Guard this file. Store a copy outside the server (USB drive, password manager). If you lose this, you lose access to your funds.

2. Prepare Your New Docker Image

You need your forked image available to Unraid. There are two approaches:

1

Option A: Push to a container registry (recommended)

Build and push from your development machine:

# From your machinaris project directory
# Build the image (this takes a while — installs all blockchain forks)
docker build -t your-dockerhub-user/machinaris:latest -f docker/dockerfile .

# Push to DockerHub (or GitHub Container Registry)
docker push your-dockerhub-user/machinaris:latest

# Or for GHCR:
docker tag your-dockerhub-user/machinaris:latest ghcr.io/your-github-user/machinaris:latest
docker push ghcr.io/your-github-user/machinaris:latest
2

Option B: Build directly on Unraid

If you prefer to build on the Unraid server itself:

# SSH into your Unraid server
ssh root@your-unraid-ip

# Clone your fork
cd /mnt/user/appdata
git clone https://github.com/SquareMesh/machinaris.git machinaris-src
cd machinaris-src

# Build locally (no push needed)
docker build -t machinaris-custom:latest -f docker/dockerfile .
Warning Building on Unraid can take significant time and CPU. The build installs 34+ blockchain forks. Consider building on a more powerful machine and pushing to a registry instead.

3. Stop and Remove the Old Container

Important Stopping and removing a container does not delete your data. Your plots, wallet, blockchain database, and configuration all live on the mapped host volumes (e.g., /mnt/user/appdata/machinaris/), not inside the container.
1

Stop any active plotting jobs

Open the Machinaris WebUI (http://your-unraid-ip:8926) and navigate to Plotting → Jobs. Stop or wait for any active plotting jobs to complete. Interrupting mid-plot wastes the temp space and time.

2

Stop the container

Unraid UI method: Docker tab → click the Machinaris icon → Stop

Terminal method:

docker stop machinaris

Wait for the container to fully stop. The Chia blockchain services need a moment to shut down cleanly.

3

Remove the old container (keep the template)

Unraid UI method: Click the Machinaris icon → Remove → confirm. When prompted, choose to remove the container only (do NOT remove the image or the template if you want a fallback).

Terminal method:

docker rm machinaris
Note Unraid stores container templates in /boot/config/plugins/dockerMan/templates-user/. If you want to keep the original template as a backup (recommended), just leave it there. You can always recreate the old container from it if something goes wrong.

4. Create the New Container

Now create a new container pointing to your forked image, using the same volume mappings, ports, and environment variables as the original.

1

Method A: Unraid Docker UI (recommended)

Go to the Docker tab → click Add Container (or edit the existing template if you kept it).

Fill in the settings using the values you recorded in Section 1:

FieldValue
Namemachinaris
Repositoryyour-dockerhub-user/machinaris:latest
(or machinaris-custom:latest if built locally)
Network TypeSame as original (typically bridge)

Then add all port mappings, volume paths, and environment variables exactly as they were on the original container. The key principle: only the Repository field changes.

2

Method B: Docker command line

If you prefer the terminal, use the values from your docker inspect backup:

docker run -d \
  --name=machinaris \
  --restart=unless-stopped \
  -p 8926:8926 \
  -p 8927:8927 \
  -p 8444:8444 \
  -e TZ="Your/Timezone" \
  -e blockchains="chia" \
  -e mode="fullnode" \
  -e worker_address="YOUR_UNRAID_IP" \
  -e controller_host="YOUR_UNRAID_IP" \
  -v /mnt/user/appdata/machinaris/chia:/root/.chia \
  -v /mnt/user/plots:/plots \
  -v /mnt/cache/plotting:/plotting \
  your-dockerhub-user/machinaris:latest
Adapt this command
  • Replace YOUR_UNRAID_IP with your server's actual IP address
  • Replace Your/Timezone with your timezone (e.g., America/New_York)
  • Add all your plot volume paths (-v /mnt/disk1/plots:/plots2, etc.)
  • Add any GPU passthrough flags if you use GPU plotting
  • Match every environment variable from your original container
3

Start the container

Unraid UI: Click Apply on the container config page. The container will pull/start automatically.

Terminal: The docker run command above starts it immediately.

First startup takes a few minutes as the entrypoint script initializes blockchain services, runs database migrations, and starts the WebUI/API servers.

5. Verify the Migration

1

Check container is running

docker ps | grep machinaris

# Expected: container is "Up" with correct port mappings
2

Check container logs for errors

# Watch startup in real-time
docker logs -f machinaris

# Look for:
#   - "Starting Machinaris..." messages
#   - Gunicorn workers booting
#   - No Python tracebacks or permission errors
# Press Ctrl+C to stop following
3

Open the WebUI

Navigate to http://your-unraid-ip:8926 in your browser.

You should see the Machinaris dashboard with your existing farm data.

6. Migrate Worker Containers (if applicable)

If you run additional Machinaris containers as harvesters or plotters, repeat the same process for each one:

1

Same process, per worker

  • Document the worker's settings (docker inspect worker-name)
  • Stop and remove the worker container
  • Create a new container with the same settings but your new image
  • Verify the worker appears on the controller's Workers page
2

Verify worker connectivity

# From the controller, check workers are reporting in
curl http://your-unraid-ip:8927/workers/

# Each worker should show status "Responding" within 2-4 minutes
Note Workers fetch SSL certificates from the controller on startup. If the controller was replaced first, the certificates will be regenerated and workers may need to re-fetch them. This happens automatically on worker container restart — just ensure the controller is fully running before starting workers.

7. Troubleshooting

ProblemLikely CauseFix
WebUI not loading Port mapping missing or Gunicorn failed to start Check docker logs machinaris for errors. Verify port 8926 is mapped.
Wallet shows 0 balance Blockchain not synced yet or wallet not synced Wait for blockchain sync (can take hours). Check Blockchains page for sync status.
Plots not showing Volume paths not mapped correctly Run docker exec machinaris ls /plots — if empty, volumes are wrong. Recreate container with correct -v mappings.
"Setup" page appears /root/.chia volume not mapped (no keys found) Stop container, fix the volume mapping for /root/.chia to your appdata path, restart.
Permission denied errors File ownership mismatch docker exec machinaris chown -R root:root /root/.chia
Database migration errors Schema version mismatch docker exec machinaris bash /machinaris/scripts/setup_databases.sh reset — this recreates databases (stats history lost, farm data re-collected).
Workers not connecting Certificate mismatch or wrong controller_host Restart worker containers after controller is fully running. Check worker_address and controller_host env vars.
Container keeps restarting Entrypoint validation failure Check docker logs machinaris. Common causes: missing worker_address, spaces in hostname, mnemonic.txt is a directory (Portainer race condition).

8. Rollback Plan

If something goes wrong and you need to revert to the original image:

1

Stop and remove the new container

docker stop machinaris && docker rm machinaris
2

Restore from backup (if needed)

# Only if the new container corrupted your appdata
rm -rf /mnt/user/appdata/machinaris
cp -r /mnt/user/appdata/machinaris-backup-YYYYMMDD /mnt/user/appdata/machinaris
3

Recreate with original image

Use the saved template in Unraid's Docker tab, or recreate manually with the original image name (ghcr.io/guydavis/machinaris:latest).
